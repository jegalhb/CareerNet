# 사용자 요구사항 정의서

## 사용자 유형

| 사용자 | 설명 |
| --- | --- |
| 비회원 사용자 | 서비스 탐색, 직업 정보 조회, 회원가입/로그인 진입 |
| 로그인 사용자 | 검사 저장, 관심 직업 저장, 멘토링 신청, 마이페이지 이용 |
| 진로 탐색 사용자 | 검사 결과 기반 추천 직업과 로드맵 확인 |
| 멘토링 희망 사용자 | 관심 직업과 연결된 현직자에게 멘토링 신청 |
| 운영자/관리자 후보 | 추후 직업, 멘토, 채용공고, 커뮤니티 데이터 관리 |

| ID | 요구사항 | 우선순위 | 구현 상태 |

| FR-03 | 사용자는 진로 검사를 제출할 수 있어야 한다. | 높음 | 구현 |
| FR-04 | 사용자는 검사 결과와 추천 직업을 조회할 수 있어야 한다. | 높음 | 구현 |
| FR-05 | 사용자는 직업 목록과 직업 상세 정보를 조회할 수 있어야 한다. | 높음 | 구현 |
| FR-06 | 사용자는 직업별 관련 멘토를 확인할 수 있어야 한다. | 높음 | 구현 |

| FR-09 | 사용자는 채용공고 목록과 상세 정보를 조회할 수 있어야 한다. | 중간 | 구현 |
| FR-10 | 사용자는 커뮤니티 게시글을 조회할 수 있어야 한다. | 중간 | 구현 |
| FR-11 | 로그인 사용자는 마이페이지에서 개인 로드맵을 확인할 수 있어야 한다. | 중간 | 구현 |
| FR-12 | 비로그인 사용자는 개인 로드맵 대신 서비스 안내 UI를 볼 수 있어야 한다. | 중간 | 구현 |

## 비기능 요구사항

| ID | 요구사항 | 설명 |
| --- | --- | --- |
| NFR-01 | REST API 기반 통신 | 프론트와 백엔드는 HTTP REST API로 통신한다. |
| NFR-02 | 계층 분리 | Controller, Service, Repository를 분리한다. |
| NFR-03 | DB 마이그레이션 | Flyway로 스키마와 seed 데이터를 관리한다. |
| NFR-04 | 비밀번호 보호 | 비밀번호는 BCrypt 해시로 저장한다. |
| NFR-05 | 테스트 가능 데이터 | 목업이 아닌 MySQL seed 데이터로 화면 테스트가 가능해야 한다. |
| NFR-06 | 발표 자료화 가능성 | README, ERD, 화면설계서, 사용자 흐름도를 파일로 관리한다. |

## 향후 요구사항 후보

- JWT 기반 인증과 권한 분리
- 관리자 페이지 CRUD
- 멘토링 신청 승인/반려 상태 관리
- 커뮤니티 게시글 작성/수정/삭제
- 채용공고 검색 조건 고도화
- 검사 종류 확장 및 종합 추천 점수 산출
- 실제 외부 직업/채용 Open API 연동
